## 一、需求分析

### 1. 问题描述

本系统模拟豆瓣等电影评分与评论平台，面向管理员和普通注册用户提供电影资料维护、演员导演资料维护、电影评分、电影评论、条件查询和热门电影推荐等功能。系统以电影为核心对象，围绕电影基本信息、演员参演关系、导演执导关系、用户评分和用户评论组织数据，并通过数据库约束、视图、存储过程和触发器保证数据一致性和功能完整性。

系统采用本地 Web 方式实现，前端页面提供可视化交互界面，后端提供 REST 风格接口。课程设计数据库脚本采用 MySQL 实现，包含建库建表、完整性约束、索引、视图、存储过程、触发器和测试数据。为了便于本地演示，应用程序运行时使用 JSON 文件作为轻量持久化数据源，数据库设计与 SQL 实现脚本单独提供。

### 2. 系统功能描述

系统分为管理员和普通用户两类角色。

管理员功能包括：

1. 登录系统并进入后台界面。
2. 维护电影基本信息，包括电影名称、发行年份、时长、类型、语言、国家地区、简介和海报地址。
3. 维护演员基本信息，包括姓名、性别、出生日期、国籍、头像和简介。
4. 维护导演基本信息，包括姓名、性别、出生日期、国籍和简介。
5. 维护用户信息，包括用户名、姓名、角色、邮箱和账号状态。
6. 为电影设置演员和导演关联关系。
7. 查看电影评分、评论、类型统计、热门推荐和用户活跃统计。

普通用户功能包括：

1. 登录系统并查看电影列表。
2. 按电影名、类型、演员、导演、最低评分等条件组合查询电影。
3. 查看电影详情、演员、导演、评分和评论信息。
4. 对电影进行评分，同一用户对同一电影只能保留一条评分记录。
5. 对电影发表评论。
6. 查看热门电影推荐和统计报表。

### 3. 数据需求

系统需要存储的数据包括：

1. 用户数据：用户编号、用户名、密码哈希、真实姓名、角色、邮箱、状态、创建时间。
2. 电影数据：电影编号、电影名称、发行年份、时长、类型、语言、国家地区、简介、海报地址、平均评分、评分人数、创建时间。
3. 演员数据：演员编号、姓名、性别、出生日期、国籍、头像地址、简介。
4. 导演数据：导演编号、姓名、性别、出生日期、国籍、简介。
5. 电影演员关系数据：电影编号、演员编号、角色名。
6. 电影导演关系数据：电影编号、导演编号。
7. 评分数据：评分编号、电影编号、用户编号、评分、创建时间、更新时间。
8. 评论数据：评论编号、电影编号、用户编号、评论内容、创建时间、更新时间。

### 4. 安全性与完整性要求

1. 系统必须通过登录后使用，未登录用户不能访问业务接口。
2. 管理员可以维护电影、演员、导演和用户数据，普通用户只能查询、评分和评论。
3. 数据库表必须设置主键，保证每条记录具有唯一标识。
4. 电影与演员、电影与导演、评分与电影、评分与用户、评论与电影、评论与用户之间必须设置外键。
5. 电影评分取值范围为 0 到 10，发行年份范围为 1900 到 2100，电影时长必须大于 0。
6. 用户名唯一，同一用户对同一电影只能评分一次。
7. 用户新增、修改评分后，电影平均评分和评分人数需要自动更新。
8. 查询功能需要建立适当索引，提高电影类型、发行年份、评分、演员姓名、导演姓名等查询效率。

### 5. 社会、法律等因素

系统应避免收集与电影评分无关的敏感个人信息。用户密码不应明文保存，应保存哈希值。评论内容应遵守法律法规，不得包含违法、不良或侵犯他人权益的信息。系统在设计时预留账号禁用状态，便于管理员处理违规用户。电影图片和演员图片在正式使用时应注意版权合规，本课程设计中使用本地 SVG 占位图作为演示资源。

## 二、概念结构设计

### 1. 实体说明

系统主要实体包括用户、电影、演员、导演、评分和评论。

用户实体表示系统注册用户。电影实体表示可被浏览、评分和评论的电影。演员实体表示参演电影的人员。导演实体表示执导电影的人员。评分实体表示用户对电影给出的数值评分。评论实体表示用户对电影发表的文字评价。

### 2. 实体联系

1. 一个用户可以对多部电影评分，一部电影可以被多个用户评分，用户与电影之间通过评分实体形成多对多联系。
2. 一个用户可以对多部电影发表评论，一部电影可以拥有多条评论，用户与电影之间通过评论实体形成一对多到多对一联系。
3. 一部电影可以有多个演员，一个演员可以参演多部电影，电影与演员之间是多对多联系。
4. 一部电影可以有多个导演，一个导演可以执导多部电影，电影与导演之间是多对多联系。

### 3. E-R 模型文字说明

电影是系统核心实体。电影通过电影演员关联表连接演员，通过电影导演关联表连接导演。用户通过评分表和评论表参与电影互动。评分表除记录用户和电影编号外，还记录评分值；评论表记录用户、电影和评论内容。电影表中的平均评分和评分人数由评分表通过触发器维护，避免应用程序手工计算造成不一致。

## 三、逻辑结构设计

### 1. 关系模式设计

用户表：

`users(user_id, username, password_hash, real_name, role, email, status, created_at)`

电影表：

`movies(movie_id, title, release_year, duration, genre, language, country, synopsis, poster_url, rating, rating_count, created_at)`

演员表：

`actors(actor_id, name, gender, birth_date, nationality, photo_url, bio)`

导演表：

`directors(director_id, name, gender, birth_date, nationality, bio)`

电影演员关联表：

`movie_actor(movie_id, actor_id, role_name)`

电影导演关联表：

`movie_director(movie_id, director_id)`

评分表：

`ratings(rating_id, movie_id, user_id, score, created_at, updated_at)`

评论表：

`comments(comment_id, movie_id, user_id, content, created_at, updated_at)`

### 2. 规范化说明

系统各表均以主键唯一标识记录。电影、演员、导演、用户等实体属性仅依赖对应主键，不存在部分依赖。电影演员、电影导演关系分别拆分为独立关联表，避免在电影表中保存重复演员或导演字符串。评分和评论独立建表，避免用户互动数据与电影基本信息混杂。因此主要关系模式满足第三范式要求。

### 3. 子模式设计

系统设计两个视图。

`v_movie_summary`：用于汇总电影基本信息、演员列表、导演列表、评分人数和评论数量，便于电影列表页面一次性展示综合信息。

`v_hot_movies`：基于电影平均评分、评分人数和评论数量计算热门分值，用于热门电影推荐。

## 四、数据库的物理设计

为了提高查询效率，系统创建以下索引：

1. `idx_movies_query`：建立在电影类型、发行年份、评分字段上，用于电影组合查询。
2. `idx_movies_title`：建立在电影名称字段上，用于电影名模糊查询。
3. `idx_actors_name`：建立在演员姓名字段上，用于按演员查询电影。
4. `idx_directors_name`：建立在导演姓名字段上，用于按导演查询电影。
5. `idx_comments_movie_time`：建立在评论电影编号和创建时间字段上，用于按电影查看评论。
6. `idx_ratings_movie_score`：建立在评分电影编号和评分字段上，用于评分统计。

这些索引覆盖了电影列表筛选、演员导演检索、评论展示、评分统计等常见访问路径。

## 五、数据库设计实现及运行

### 1. 数据库创建

数据库名为 `movie_rating_system`，字符集为 `utf8mb4`，排序规则为 `utf8mb4_unicode_ci`，支持中文和常见特殊字符。

完整 SQL 位于 `sql/01_schema.sql`。

### 2. 数据表创建

系统创建了用户、电影、演员、导演、电影演员关联、电影导演关联、评分和评论八类数据表。各表均使用 InnoDB 存储引擎，以支持事务和外键约束。

### 3. 视图创建

创建 `v_movie_summary` 和 `v_hot_movies` 视图。前者用于电影列表综合展示，后者用于热门电影推荐。

### 4. 存储过程定义实现

系统创建以下存储过程：

1. `sp_movie_comments(IN p_title VARCHAR(120))`：以电影名为输入参数，查询匹配电影的评论信息。
2. `sp_actor_movies(IN p_actor_name VARCHAR(60))`：以演员姓名为输入参数，查询该演员参演的电影信息。
3. `sp_genre_report()`：按电影类型汇总电影数量、平均评分和评分次数。

### 5. 触发器定义实现

系统创建三个评分触发器：

1. `trg_ratings_after_insert`：新增评分后自动更新电影平均评分和评分人数。
2. `trg_ratings_after_update`：修改评分后自动更新电影平均评分和评分人数。
3. `trg_ratings_after_delete`：删除评分后自动更新电影平均评分和评分人数。

### 6. 数据操纵语句

系统在 `sql/02_seed.sql` 中插入了管理员、普通用户、电影、演员、导演、电影演员关系、电影导演关系、评分和评论测试数据。应用程序通过接口完成增、删、改、查操作。

## 六、系统详细设计及实现

### 1. 系统结构

系统采用本地 Web 架构。服务端由 `server.js` 实现，使用 Node.js 标准库创建 HTTP 服务，负责静态资源访问、登录认证、权限检查、业务接口和 JSON 数据持久化。前端由 `public/index.html`、`public/styles.css` 和 `public/app.js` 实现，提供登录、电影管理、演员管理、导演管理、用户管理、报表统计和存储过程演示页面。

### 2. 登录模块

登录模块根据用户名和密码哈希校验用户身份。登录成功后服务端生成随机 token，并在后续请求中通过 `Authorization: Bearer token` 识别当前用户。普通用户和管理员在界面上看到的操作权限不同。

### 3. 电影模块

电影模块支持电影列表查询、电影详情查看、管理员新增电影、编辑电影和删除电影。电影查询支持电影名称、类型、演员、导演、最低评分和排序方式等条件。电影详情页展示电影基本信息、演员、导演、评分和评论。

### 4. 评分评论模块

用户可以在电影详情页提交评分和评论。评分提交后，服务端会更新当前用户对该电影的评分记录，并重新计算电影平均分和评分人数。评论提交后，电影详情页会立即刷新评论列表。

### 5. 演员导演模块

管理员可以维护演员和导演资料，普通用户可以查看演员和导演信息。删除演员或导演时，如果该人员已经被电影引用，系统会阻止删除，以保证关联关系完整。

### 6. 用户管理模块

管理员可以新增和编辑用户，设置用户角色和启用状态。普通用户不能访问用户管理页面。

### 7. 报表统计模块

系统提供热门电影推荐、电影类型评分汇总、用户活跃统计，以及模拟存储过程调用的电影评论查询和演员参演电影查询。报表页面可用于答辩时展示视图和存储过程对应的业务效果。

### 8. 关键文件

1. `server.js`：后端服务和业务接口。
2. `public/app.js`：前端页面渲染、表单校验和 API 调用。
3. `data/db.json`：本地演示数据。
4. `sql/01_schema.sql`：数据库结构、视图、存储过程和触发器。
5. `sql/02_seed.sql`：测试数据和存储过程调用。

## 七、总结

本课程设计围绕电影评分系统完成了从需求分析、概念结构设计、逻辑结构设计、物理设计到系统实现的完整过程。通过本次设计，我进一步理解了数据库应用系统中实体关系建模、关系模式设计、完整性约束、视图、存储过程和触发器的作用，也体会到前后端系统与数据库设计之间需要保持一致。

系统基本实现了电影资料维护、演员导演资料维护、用户登录、评分、评论、组合查询、热门推荐和报表统计等功能。数据库部分设计了主键、外键、CHECK 约束、索引、视图、存储过程和触发器，能够满足课程设计对数据库对象的要求。

目前系统仍有可以改进的地方，例如可以进一步接入真实 MySQL 数据库驱动，将应用运行数据直接写入 MySQL；可以增加评论审核、电影图片上传、分页查询和更复杂的推荐算法。总体而言，本系统达到了课程设计预定目标，能够作为数据库应用课程设计的演示系统。
